loading in iOS production build (works fine in Expo Go).

DIAGNOSIS NEEDED FIRST:
Run this search to find where transliteration and 
translation data is loaded in the Quran tab:

grep -rn "transliteration\|translation\|quran-api\|quranApi\|fetchTranslit\|loadTranslit" \

  --include="*.tsx" --include="*.ts"

════════════════════════════════════════════════════════
LIKELY CAUSES AND FIXES
════════════════════════════════════════════════════════

CAUSE A — Data is fetched from an API URL (not bundled):
If transliteration or translation is fetched from 
alquran.cloud, api.quran.com, or any external URL,
it WILL fail in production without internet.

FIX A: Check lib/quran-api.ts for any fetch() calls.
If found, the data must be bundled locally instead.
Download all transliteration data and embed it as 
a JSON constant in constants/ folder.

CAUSE B — Large JSON files not included in iOS bundle:
If transliteration/translation data is stored in 
.json files in assets/ or constants/, they may not 
be included in the iOS production bundle.

FIX B: In app.json, make sure all data files are 
explicitly included:

"expo": {
  "assetBundlePatterns": [
    "**/*",
    "assets/**/*",
    "constants/**/*.json"
  ]
}

CAUSE C — Dynamic require() not resolved in production:
If the code uses dynamic requires like:
  const data = require(`../constants/${lang}-quran.json`)
These work in Expo Go but fail in production builds.

FIX C: Replace all dynamic requires with static imports:
  import enData from '../constants/en-quran.json'
  import frData from '../constants/fr-quran.json'
  // etc.
  const translations = { en: enData, fr: frData, ... }

CAUSE D — File too large for metro bundler:
Files over ~50MB may be silently excluded from 
production builds.

FIX D: Check file sizes:
  find constants/ assets/ -name "*.json" -exec ls -lh {} \;

If any single file is over 30MB, split it by language.
